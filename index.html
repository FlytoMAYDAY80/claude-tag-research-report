01核心结论

Claude Tag 是 Anthropic 于 2026 年 6 月 23 日发布的团队协作 Agent，首发于 Slack，面向 Claude Team 与 Enterprise 用户开放 Beta。它会替代原有 Claude in Slack 体验，官方帮助中心给出的切换日期是 2026 年 8 月 3 日。

产品范式
Multiplayer

同一频道成员与同一个 Claude 协作；任务、进展与结果对团队可见，其他人可以接续。

核心跃迁
4 个变化

共享记忆、主动介入、长任务异步执行、独立 Agent 身份与审计。

当前范围
Slack only

首发仅 Slack，使用 Opus 4.8；Anthropic 表示未来会扩展到其他团队工作场所。

最重要的理解：旧版是“人在需要时调用 Claude”；Claude Tag 则是“把一个有权限、有记忆、会持续工作的 Claude 放进频道”。界面动作仍是 @Claude，但底层协作模型已经不同。

02功能定位

Claude Tag 的官方定位是“让团队与 Claude 一起工作的新方式”。从产品结构看，它更准确的类别是：嵌入协作空间的、组织级、多人共享的通用执行 Agent。

它是什么

  • Slack 频道里的共享 AI 队友，而非个人聊天助手。
  • 以组织配置的工具、数据、代码库和 Skills 执行任务。
  • 保留频道/工作区记忆，可跨线程、跨天延续工作。
  • 可被 @ 唤起，也可按 standing instructions 主动运行。

它不只是

  • 不是给 Slack 加一个“问答机器人”。
  • 不是单纯把 Claude 网页版嵌进侧边栏。
  • 不是只面向编程的 Claude Code Slack 入口。
  • 也不是每个用户各自拥有、彼此隔离的私人 Agent。

典型使用场景

发现工作Bug、指标异常、客户问题或未决讨论出现在频道。
就地委派任何成员 @Claude，用自然语言说明目标。
读取上下文结合线程、频道记忆和获准的数据源理解任务。
异步执行分解步骤，调用工具，可持续数小时或数天。
回到团队在线程汇报结果、创建 PR/工单，或主动 @ 人做决定。

03产生背景

2025-10-01

Claude in Slack（初代）：把“自己的 Claude”带进 Slack

首次进入 Slack，提供双向连接：一方面可通过 DM、AI assistant panel 和线程 @Claude 随时调用 Claude；另一方面可把 Slack 连接为 Claude 的数据源，在 Claude 中搜索频道、私信和共享文件。每个用户仍以自己的 Claude 账号、个人权限和连接器工作，面对的是“自己的助手”。

2025-12-08

Claude Code in Slack：从讨论直接发起编码任务

开发者在 thread 里 @Claude，即可拉起完整的 Claude Code Web 会话。Claude 会分析最近消息中的 Bug 报告、需求和复现信息，判断应使用的代码仓库，在线完成调查、编写测试和修改代码；执行进度回贴到原 thread，完成后提供可审阅的会话与 PR 链接。AI 开始从“回答”走向“后台做完”。

2026-06-23

Claude Tag：从单次任务走向持续的团队成员

加入频道级共享记忆、主动行为、计划任务、长时运行与 Agent Identity，并从工程任务扩展到数据、客服、运营等通用工作。

2026-08-03

旧版 Claude in Slack 计划切换到 Claude Tag

原有交互入口继续存在，但频道交互将采用新的组织身份、记忆与计费模式。

产品背景分析：Claude Tag 回应的是一个结构性矛盾——工作上下文越来越多地沉淀在群聊里，而 Agent 的执行能力却长期困在“一人一聊、一问一答”的界面中。要让 AI 真正成为团队成员，必须同时补上多人协作、持续记忆、后台执行和组织权限治理四块基础设施。
内部验证
65%

Anthropic 称其产品团队目前 65% 的代码由内部版 Claude Tag 创建。该数字是厂商自述，未披露统计口径与外部审计。

为何先选 Slack
工作发生地

Anthropic 称 Slack 是人类团队与 AI 协作的自然场所，也是公司日常工作大量发生的地方。

04主要能力

@

频道内自然委派

在任意获准频道或线程里 @Claude，要求其拉数、整理结论、建工单、修 Bug 或开草稿 PR。

∞

跨线程、跨天的记忆

记住频道和工作区里的相关信息；管理员可查看、编辑和删除这些记忆。

↻

异步长任务

把工作拆成阶段并依次执行，可在数小时或数天内持续推进，并在需要输入时回来询问。

!

主动与环境式介入

Ambient 模式开启后，可主动提示相关信息、追踪沉默线程、在任务完成时回报或 @ 负责人。

⌁

工具与数据执行

可连接 API、企业数据、代码库、Skills 与插件，执行查询、写入、分析、创建产物等动作。

ID

独立 Agent Identity

共享频道内不是借用某个用户账号，而是使用管理员为 Claude 配置的服务身份、凭据和边界。

三层权限与记忆边界

层级作用范围继承关系适用示例
组织级Claude Tag 安装范围内通用凭据与代码库作为基础能力向下继承低风险知识库、通用只读工具
WorkspaceSlack 工作区内所有公共频道继承组织级权限与记忆公共项目资料、通用数据查询
私有频道单个私有频道的额外工具与仓库在工作区基础上增加隔离能力法务资料、销售 CRM、特定代码仓

补充：频道内工作由组织付费；DM 仍使用用户自己的 Claude 账号、个人连接器和个人计费。频道任务按使用量计费，可设置组织总额度、频道额度及 75%/95% 预警。

05核心优势

01 / Context

上下文复利

团队不再为每次提问重述背景。频道记忆、其他获准频道和外部系统组合后，Claude 对组织的理解会随使用累积。

02 / Workflow

决策点即执行点

问题出现在哪里，就在哪里委派；减少从 Slack 复制上下文、切换到 Agent、再搬回结果的损耗。

03 / Team

真正多人协作

任务不是锁在发起人的私人会话里。整个频道能看进度、补充信息、改变方向并接力后续工作。

04 / Autonomy

从响应到主动

Standing instructions、调度和 Ambient 让 Claude 不只回答当下问题，还能盯任务、跑例行工作、主动追踪异常。

05 / Execution

通用任务 + 编码深度

继承 Claude Code 的长任务与代码执行优势，同时扩展到指标、客服、会议准备、工单处理和跨系统工作。

06 / Governance

面向 Agent 的治理模型

独立身份、频道隔离、网络白名单、审计日志和成本上限，使共享 Agent 不必借用个人凭据运行。

真正的护城河不只是模型：Claude Tag 将“模型能力、频道上下文、组织记忆、工具连接、长时任务、权限审计”打包成一套协作运行时。竞争的单位从单次回答质量，变成 Agent 能否长期嵌入团队工作系统。

06与之前 Slack 里 @Claude 的差别

维度原 Claude in Slack / Claude Code in SlackClaude Tag
本质定位个人 Claude 的 Slack 入口；遇到编码请求时可路由到 Claude Code。驻留在频道中的组织级共享 Agent。
触发方式DM、侧边栏、明确 @ 后响应。保留 DM/侧边栏/@；另支持 standing instructions、计划任务和主动介入。
频道上下文频道 @ 读取最近 20 条消息；线程 @ 读取最近 50 条；转发线程最多 100 条。跨线程、跨天保留频道/工作区记忆；官方未公开固定数字窗口。
记忆归属每个用户/会话各自维护上下文；频道不是共享长期记忆主体。记忆归属于频道与工作区边界，管理员可查看、编辑、删除。
多人接力回答可见，但任务本身主要由发起人的会话驱动。频道里的所有人面对同一个 Claude，可共同 steering、接续和复用。
任务持续时间普通问答为同步响应；编码任务可在 Claude Code Web 后台运行。通用任务均可拆解并异步运行，支持持续数小时或数天及自我调度。
主动性主要由用户发起。可主动发布完成状态、追踪未决事项、提示跨频道信息并 @ 负责人。
身份与凭据用户认证自己的 Claude 账号，并使用个人连接器与用户权限。频道任务使用组织配置的 Agent Identity；DM 仍使用个人身份。
权限边界“发起用户能访问什么，Claude 就能访问什么”。“这个频道里的 Claude 被允许做什么”；公共 workspace 与私有频道可隔离。
审计与成本依赖个人账号使用限额与各系统日志。组织/频道额度、阈值告警、任务与网络调用审计；频道按组织用量计费。
迁移关系旧体验。官方明确表示会替代旧版；计划于 2026-08-03 切换。

用一句类比概括

旧版 @Claude 像“把一个很强的外部顾问临时拉进电话”；Claude Tag 像“给团队增加一位有工号、有权限、有记忆、会主动追活、还能夜里继续干活的同事”。

07Claude 三代界面演进

界面上仍然都是“在群聊里 @ 一个 AI”，但产品角色已经从即时回答的 App，演进为会展示执行进度、调用外部工具、持续完成任务的 Agent。

2025 年 10 月 · Claude in Slack（初代）

这一代的关键是“双向连接”：既把 Claude 带入 Slack，随时可在 DM、侧边栏或 thread 调用；也把 Slack 变成 Claude 可检索的数据源。但权限、连接器、会话与结果发布仍围绕单个用户组织，每个人面对的是“自己的助手”。

2025 年 12 月 · Claude Code in Slack

这是承上启下的一代：交互入口仍是旧版 @Claude，但被触发的不再只是一次回答，而是一个有独立运行过程的编码 Session。它验证了“从群聊讨论直接委派后台工作”的价值，也成为 Claude Tag 向通用任务扩展的前身。

2026 年 6 月 · Claude Tag：新版频道 Agent

三代演进一眼看懂：初代是“Claude 给我一个回答”；Claude Code in Slack 是“把这一条工程任务交给后台编码 Session”；Claude Tag 则是“频道共同拥有一个有记忆、有权限、可主动、可持续工作的通用 Agent”。真正的变化不在 @ 的样式，而在 @ 之后启动的工作系统。

08有没有类似产品？

有，而且赛道正在快速收敛到“Agent 进入群聊/频道”这一形态。但各家相似的是入口，不一定相似的是记忆、身份与长期自治能力。

ChatGPT Agents in Slack

OpenAI · 把可定制的 workspace agents 部署到 Slack 频道

最接近的直接竞品
进入哪个协作平台

当前进入 Slack 公共/私有频道；Slack Enterprise Grid 可跨多个 workspace 使用。Agent 先在 ChatGPT 工作区中创建，再分配 @handle 与目标频道。

怎样响应用户请求

频道可配置“仅在 @mentioned 时回复”或“读取相关消息并自行决定回复”；被提及后还能继续跟进线程里的相关回复，并支持按 schedule 主动向频道发消息。

平台内外数据与工具

可使用 Agent Builder 中配置的 Tools、Skills、Files 与 Memory；官方示例展示 Gmail、Notion 和内部文件，并说明可连接私有系统、发邮件、更新文档及向 Slack 输出文件。

企业管理员怎么管理

Slack 侧需批准应用；ChatGPT Enterprise 管理员还需开启 Workspace agents in Slack，并可用 RBAC 控制访问。Enterprise Grid 需要组织级批准、workspace 级安装和成员连接三层配置。

与 Claude Tag 的关键差别：ChatGPT 更像“企业自己创建一组专用 Agent，再分别投放到频道”；Claude Tag 更像一个统一的共享 @Claude。OpenAI 当前公开材料未明确承诺 Claude Tag 式的频道共享长期记忆、数天长任务和独立 service identity 审计。
官方资料：ChatGPT Agents App in Slack

Agentforce in Slack

Salesforce · CRM 与员工服务驱动的业务 Agent 平台

业务系统最强近邻
进入哪个协作平台

Agent 在 Salesforce Agentforce Builder 中创建，再部署到 Slack；用户可从 Agents 标签页、DM 或频道中访问，像添加同事一样把 Agent 放进工作流。

怎样响应用户请求

加入频道后通过 @mention 响应，也可被 Slack Workflow 触发。官方模板覆盖 Customer Insights、Employee Help 和 Onboarding，可回答、查数据并继续执行 CRM/Slack 动作。

平台内外数据与工具

结合 Slack 消息/文件、Salesforce Customer 360 与 Data Cloud Library；Enterprise Search 可接 Google Drive、OneDrive、Asana 等第三方系统。Slack Actions 可搜索、建 Canvas、建频道、发 DM。

企业管理员怎么管理

需先连接 Salesforce org 与 Slack、映射用户，再由 Slack owner/admin 审核 Agent 权限并安装到指定 workspace。管理员可管理 Agent profile、成员访问与卸载；Service Agent 默认无人可用，必须显式授权。

与 Claude Tag 的关键差别：Agentforce 是“围绕 Salesforce 业务对象搭建多种专用 Agent”，优势是 CRM 与流程深度；回答遵循发起者的 Slack/Salesforce 用户权限，而非 Claude Tag 的频道 Agent Identity。
官方资料：Set up and manage Agentforce in Slack · Agentforce for Employees

Glean in Slack

Glean · 企业知识检索、权限感知问答与后台 Agent

知识上下文最强
进入哪个协作平台

通过 Glean Slack App/Gleanbot 进入 Slack 频道、DM 侧边栏与搜索入口；也可把 Slack 作为 Glean 的实时搜索和企业知识数据源。

怎样响应用户请求

可在频道被 @Glean 时回答，或在检测到问题后自动介入；还提供 /glean 搜索、个人 Daily Digest、主动频道/线程摘要，以及定时或事件触发的后台 Agent 消息。

平台内外数据与工具

检索有权限的 Slack 公共/私有频道、DM 与文件，并结合 100+ 企业应用。Agent 可向 Slack 发/定时发消息、创建频道、邀请成员、设提醒；Code Writer 还能从讨论生成草稿 PR。

企业管理员怎么管理

管理员在 Glean Admin Console 添加并授权 Slack 数据源、启用 Events 与 Gleanbot，选择自动回答范围并管理生成式 AI 成本。搜索与回答继承原系统权限，管理员还可管理 Actions、Agent Builder 与数据治理。

与 Claude Tag 的关键差别：Glean 的中心是“权限感知的企业知识与 system of context”，强在找准资料、主动摘要和跨源检索；它未公开提供 Claude Tag 式的频道共享长期身份与统一 Agent 记忆。
官方资料：Glean in Slack · Glean Help Center · Slack Actions

Dust in Slack

Dust · 模型中立、可配置的公司 Agent 平台

定制最灵活
进入哪个协作平台

@Dust 当前支持 Slack 公共与私有频道，不支持 DM。企业可把一个专用 Dust Agent 绑定到特定频道，例如让 #hr-questions 默认由 @hr_help 响应。

怎样响应用户请求

用户用 @Dust +agent 调用指定 Agent，或直接 @Dust 触发频道默认 Agent；也可借助 Slack Workflow 自动触发。回复带入被提及的 thread 上下文并公开给频道成员。

平台内外数据与工具

可连接 Google Drive、Notion、Confluence、GitHub、Microsoft、Zendesk、Snowflake、BigQuery、Gong 等数据源，并通过原生工具或 MCP 使用个人/共享凭据。Slack 触发本身只看到该 thread，而非完整频道历史。

企业管理员怎么管理

管理员启用 Slack integration 并批准权限；在 Dust 侧统一管理用户与角色、Spaces、Agents、Connections、Tools/MCP、审计日志和 Credits。Dust 还刻意不把 Agent 自己的 Slack 回答重新同步为知识源，避免错误循环传播。

与 Claude Tag 的关键差别：Dust 更像一个可以组装许多 Agent 的中立平台，灵活但需要企业自己设计数据、工具、触发和记忆策略；Slack 端默认只有线程上下文，缺少 Claude Tag 的跨线程频道记忆。
官方资料：Dust in Slack · Slack Troubleshooting

Microsoft 365 Copilot in Teams group chats

Microsoft · Teams 群聊参与者与 Microsoft 365 Agent 生态

M365 原生近邻
进入哪个协作平台

Copilot 可作为参与者加入 Microsoft Teams 一对一或群聊；Microsoft 365 Agents 还可分发到 Microsoft 365 Copilot、Teams、Outlook 等统一应用平台。

怎样响应用户请求

群成员通过 @Copilot 提问；如果答案仅使用 Web 或所有成员均有权访问的来源，则直接公开。若引用了并非所有人可见的资料，发起人先收到私密预览，并决定 Approve 或 Reject 分享。

平台内外数据与工具

以发起人的权限读取其可访问的文档、SharePoint、聊天/频道历史与 Web；Microsoft 365 Copilot 还通过 Graph 使用邮件、会议、文件等工作数据。自定义 Agent 可增加 Jira、Dynamics 365、Bing、连接器、API 与 Actions。

企业管理员怎么管理

Teams Admin Center 控制 Copilot App 的访问和固定；Microsoft 365 Admin Center 的 Copilot Control System 可盘点 Agent、查看数据源与自定义动作，并按用户/组部署、阻止、移除或审批。Copilot Studio 还可通过 Power Platform DLP 限制连接器与发布渠道。

与 Claude Tag 的关键差别：Microsoft 的核心边界是“谁发起提问，就用谁的 Microsoft 365 权限和数据”；Claude Tag 的核心是“频道里的 Claude 有一套独立于任何个人的 Agent Identity”。
官方资料：Copilot in Teams group chats · Manage agents for Microsoft 365 Copilot · Copilot Studio data policies

09局限与风险

Beta 与平台范围

目前只在 Slack、Team/Enterprise Beta 提供；迁移窗口短，能力和管理界面仍可能变化。

新权限语义

只要频道的 Agent profile 获准，频道成员可能让 Claude 读取其本人没有直接权限的仓库。这是设计特征，也要求按最小权限配置频道。

主动误判与噪音

Ambient 如果过度积极，会产生打扰、错误升级或“自动追活”压力。因此默认关闭，应该从低风险频道灰度。

记忆污染

错误结论若被写入长期记忆，可能在后续任务中持续放大。管理员需要定期查看、纠正与删除记忆。

用量成本

按使用量而非席位计费，且使用 Opus 4.8。长任务、多频道与主动模式会带来成本波动，必须设置频道额度。

厂商指标待验证

“65% 产品团队代码由内部版生成”是 Anthropic 自报，缺少口径与独立验证，不应直接外推为客户 ROI。

10选型与落地建议

怎么选

  • 选 Claude Tag：团队主要在 Slack 工作，需要一个通用、高能力、能持续数小时/数天并深度处理代码的共享 Agent。
  • 选 ChatGPT Agents：核心诉求是让业务团队自行创建多个专用 Agent，并已使用 ChatGPT 企业工作区。
  • 选 Agentforce：关键动作围绕 Salesforce CRM、客服与员工服务，业务对象和流程深度优先。
  • 选 Glean：首要难题是企业知识散落与权限感知检索，执行自动化是第二优先级。
  • 选 Dust：希望模型中立、快速组装多个定制 Agent，并愿意自己搭建工作流与记忆策略。
  • 选 Microsoft Copilot：组织协作主阵地是 Teams/Microsoft 365，而不是 Slack。

Claude Tag 建议的试点顺序

1. 低风险频道从内部公开、非敏感、可人工复核的频道开始。
2. 只读连接先接知识库与只读数据，暂缓高风险写入。
3. 三个任务固定测试摘要、指标查询、工单/草稿 PR。
4. 审计复盘检查记忆、网络调用、错误率、成本与噪音。
5. 再开主动验证价值后再启用 standing instructions 与 Ambient。
